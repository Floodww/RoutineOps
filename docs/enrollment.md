# Энроллмент устройства

Актуально на 2026-07-24 (v2.5.0).

Как новое устройство получает свой mTLS-сертификат «за пару шагов»: агент сам
генерирует ключевую пару и CSR, сервер подписывает CSR по enrollment-токену.
Токены двух видов: **одноразовый** (привязан к заранее созданному устройству) и
**массовый** (bulk — одна команда установки на целую партию устройств). Ниже —
принципы, флоу, REST-контракт эндпоинта `/api/v1/enroll` и обоснование принятых
решений.

## Принципы

- **Приватный ключ никогда не покидает устройство.** Агент сам генерит ключевую
  пару и CSR; сервер подписывает CSR своим CA. Сервер не генерит и не хранит
  приватные ключи агентов. (Это же готовит почву под Keychain/Cert Store.)
- **CN = device_id ставит СЕРВЕР, не CSR.** Сервер ИГНОРИРУЕТ subject/CN
  из CSR и проставляет CN сам (device_id). Скомпрометированный энроллер не может
  выбрать чужой device_id. Из CSR берётся только публичный ключ.
- **Bootstrap до mTLS.** На момент энроллмента у агента ещё нет клиентского серта,
  поэтому канал энроллмента — server-auth TLS (не mTLS), аутентификация запроса —
  по одноразовому enrollment-токену.

## Флоу

1. **UI, одиночное устройство:** IT-админ жмёт «Добавить устройство» → сервер создаёт
   запись pending-устройства с уже назначенным `device_id` и одноразовый
   `enrollment_token` (UUID v4, TTL 24ч, single-use), привязанный к этому
   device_id. Устройство видно в списке ещё до энроллмента — можно заранее задать
   hostname/теги. Админу показывается команда установки (адрес сервера + токен; CA
   несёт установочный пакет).

   **UI, массовый энроллмент (миграция 031):** админ выпускает bulk-токен — он не
   привязан к устройству (`device_id` NULL), многоразовый (лимит `max_uses` или
   безлимит до `expires_at`), может нести целевую группу и флаг `require_approval`.
   Устройство создаётся сервером **в момент редима токена**; с `require_approval`
   оно попадает в очередь одобрения (статус `pending_approval`, одобрить/отклонить —
   в UI, одобрение — только человеком). Секция «Не подключились» показывает
   одобренные устройства без единого heartbeat. Отзыв токена (кнопка или
   `DELETE /enrollment-tokens/{id}`) — мгновенный: сервер сдвигает `expires_at` в
   now(); уже заэнролленные устройства отзыв не трогает.
2. **Агент, первый запуск (`agent enroll`):**
   - генерит ECDSA P-256 ключевую пару локально;
   - строит CSR (subject можно оставить пустым — сервер всё равно переопределит CN);
   - вызывает enroll-эндпоинт с токеном + CSR + hostname/os/arch;
   - получает `device_id`, подписанный `cert_pem`, `ca_pem`;
   - раскладывает `agent.key` (локально сгенерённый), `agent.crt`, `ca.crt` в
     стандартный каталог конфигурации; опционально регистрирует службу (`install`).
3. **Сервер:** валидирует токен (есть, не истёк, не использован) → подписывает CSR
   своим CA, CN=device_id → помечает токен использованным, связывает серийник серта
   с устройством → отдаёт серт + CA.
4. **Дальше** — обычная работа агента по mTLS (Connect/heartbeat/...).

## Контракт — REST

Энроллмент — одноразовый bootstrap, ложится в REST поверх уже имеющегося
HTTP-стека сервера (порт `:8081`, без клиентского серта; через nginx доступен
и на `:443`) и не требует выкраивать в gRPC-сервере исключение «без mTLS».

```
POST /api/v1/enroll   (HTTP :8081, server-auth TLS, без mTLS)
Content-Type: application/json

{
  "enrollment_token": "<одноразовый токен>",
  "csr_pem": "-----BEGIN CERTIFICATE REQUEST----- ...",
  "hostname": "MacBook-Air.local",
  "os": "macOS",
  "arch": "arm64"
}
```

Ответ `200 OK`:
```json
{
  "device_id": "ed7433b7-...",
  "cert_pem": "-----BEGIN CERTIFICATE----- ...",
  "ca_pem": "-----BEGIN CERTIFICATE----- ...",
  "release_pubkey": "<base64 ed25519>"
}
```

`release_pubkey` — публичный ключ подписи релизов этой инсталляции: агент сохраняет
его при энроллменте и проверяет им подпись self-update-манифестов (сборки по
умолчанию универсальные, без вшитого ключа — см. [self-update.md](self-update.md)).

Ошибки: `401` — токен неизвестен/истёк/использован; `400` — битый CSR;
`409` — устройство уже заэнроллено (повторный enroll запрещён по умолчанию,
см. решение 7).

> gRPC-вариант (`EnrollmentService` на отдельном listener без mTLS) рассматривался
> и отклонён: REST на :8081 не требует второго listener / per-service TLS.

## Агентская сторона (`internal/agent/enroll`)

Подкоманда:
```
agent enroll -enroll-url https://<host>/api/v1/enroll -token <tok> \
             -ca ca.crt -cert agent.crt -key agent.key [-install-service]
```
- Генерит ECDSA P-256 ключ локально (приватный ключ не уходит) + CSR с пустым
  subject (`x509.CreateCertificateRequest`).
- HTTPS-клиент **пинит `-ca`** (CA из пакета) — доверие к enroll-эндпоинту до
  получения рантайм-CA. `POST /api/v1/enroll` с токеном + CSR + hostname/os/arch.
- Проверяет, что выданный серт подписан **под наш** публичный ключ (сервер не
  подсунул чужой), затем раскладывает `agent.key` (0600), `agent.crt`, `ca.crt`.
- При `-install-service` после успеха регистрирует службу с этими путями.
- Флаги дублируются env: `ROUTINEOPS_ENROLL_URL`, `ROUTINEOPS_ENROLL_TOKEN`, `ROUTINEOPS_ENROLL_INSTALL`.

Серверная сторона (`POST /api/v1/enroll`, подпись CSR, UI токенов) реализована
и отвечает по контракту выше: `{device_id, cert_pem, ca_pem}`; ошибки
`401`/`400`/`409`.

## Решения дизайна

1. **Транспорт — REST.** `POST /api/v1/enroll` на :8081 (HTTP уже без клиентского
   серта). gRPC отклонён — не нужен второй listener под bootstrap без mTLS.
2. **Доверие к серверу — вариант (a): CA в пакете.** Установочный пакет несёт
   `ca.crt`, агент пинит его при enroll-запросе. Токен + CA = минимальная
   доверенная база. LE не годится — CA приватная.
3. **Токен:** UUID v4, TTL 24ч, одноразовый (`used_at` при успешном enroll),
   ревокация кнопкой в UI. Генерит сервер при создании pending-устройства.
   В БД хранится только SHA-256-хеш токена (миграция 028) — plaintext показывается
   один раз при выдаче, дамп БД рабочих токенов не содержит. Bulk-токены (031) —
   исключение из одноразовости: см. «Флоу».
4. **device_id — заранее (одиночный путь).** При «Добавить устройство» создаётся
   запись в `devices` (статус pending), токен привязан к device_id. Агент получает
   уже существующий device_id. Устройство видно в списке и конфигурируется до
   энроллмента. Массовый путь работает наоборот: устройство создаётся при редиме
   bulk-токена.
5. **Установочный пакет.** Windows — MSI (делает enrollment и ставит службу сам,
   см. `docs/install.md`); Linux/macOS — сгенерированный shell-installer: скачивает
   бинарь по HTTP + сверяет sha256-пин → `agent enroll -install-service`.
6. **CSR subject игнорируется.** Сервер игнорирует CN из CSR, проставляет
   CN=device_id сам; из CSR берётся только публичный ключ.
7. **Повторный enroll — запрещён по умолчанию** → `409`, если устройство уже
   enrolled. Ре-энролл (кейс «потерял серт») — явной кнопкой в UI: сброс статуса
   в pending + выдача нового токена. Ре-энролл запрещён для устройств в статусах
   `decommissioned` / `blocked` / `rejected` / `pending_approval` — обходной двери
   в управляемые статусы через ре-энролл нет.
8. **Удаление устройства отзывает его сертификат (миграция 034).** `DeleteDevice`
   в транзакции пишет отпечаток в `revoked_fingerprints`; gRPC `Connect` отклоняет
   отозванный сертификат. Без этого живой агент удалённого устройства «воскрешал»
   его пустой записью через штатную регистрацию по неизвестному отпечатку (ADR-1).
   Ре-энролл по новому токену выпускает новый сертификат и проходит.
