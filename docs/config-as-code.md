# Конфигурация парка в YAML (`routineops` CLI)

Актуально на 2026-07-24 (v2.5.0).

Группы, скрипты, скрипт-политики и софт-правила можно держать в git и раскатывать
файлом, а не кликами. Утилита `routineops` умеет две вещи: выгрузить текущее
состояние сервера в YAML (`export`) и привести сервер к состоянию из файла
(`apply`).

Ходит она публичным HTTP API под API-токеном — теми же ручками, что и веб-интерфейс.
Отдельного канала доступа у CLI нет намеренно: иначе появился бы второй набор правил
доступа, который однажды разъедется с первым.

## Чего в файле нет

**Устройств.** YAML описывает интент — «что и кому», — а членство устройств в группах
решают энроллмент (токен несёт группу) и ручные перемещения в интерфейсе. Иначе файл
пришлось бы править на каждую новую машину, и история конфигурации в git утонула бы
в инвентаре.

Не выгружаются и не применяются также: софт-правила без группы (глобальные и
адресованные конкретному устройству), пользователи, роли, лицензия, алерты.

## Быстрый старт

```bash
make cli                       # или: go build -o bin/routineops ./cmd/routineops

export ROUTINEOPS_URL=https://mdm.example.com
export ROUTINEOPS_TOKEN=<api-токен>
export ROUTINEOPS_CA=/path/to/ca.crt        # если сервер с собственным CA

bin/routineops export -o fleet/routineops.yaml     # состояние сервера → файл
git -C fleet add -A && git -C fleet commit -m "конфигурация парка"

# правим файл руками, затем:
bin/routineops apply -f fleet/routineops.yaml -dry-run   # план без изменений
bin/routineops apply -f fleet/routineops.yaml            # применить
```

## Токен доступа

Нужен API-токен с ролью `it_admin`: `viewer` увидит выгрузку, но не применит её.

Выпустить токен может **только человек** — сервисным токеном выписать новый нельзя
(правило `requireHuman`: всё, что выдаёт или повышает права, требует живого
администратора). Отдельной страницы в интерфейсе пока нет, поэтому выпуск делается
запросом с кукой обычного входа:

```bash
curl -sk -c /tmp/c.jar -X POST https://mdm.example.com/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@example.com","password":"..."}'

curl -sk -b /tmp/c.jar -X POST https://mdm.example.com/api/v1/api-tokens \
  -H 'Content-Type: application/json' \
  -d '{"name":"gitops","role":"it_admin","expires_in_days":365}'
```

Плейнтекст токена возвращается **один раз** — в базе лежит только его SHA-256, и
список токенов его больше не покажет. `expires_in_days: 0` — бессрочный.

## Команды и флаги

| Флаг | Команда | Назначение |
|---|---|---|
| `-server` | обе | Адрес сервера, `https://mdm.example.com` (env `ROUTINEOPS_URL`) |
| `-token` | обе | API-токен (env `ROUTINEOPS_TOKEN`) |
| `-ca` | обе | CA-сертификат сервера, если он собственный (env `ROUTINEOPS_CA`) |
| `-o` | `export` | Куда писать YAML, по умолчанию `routineops.yaml` |
| `-scripts-dir` | `export` | Каталог для тел скриптов рядом с YAML, по умолчанию `scripts`. Пустое значение — складывать тела инлайном в YAML |
| `-f` | `apply` | Путь к YAML |
| `-dry-run` | `apply` | Показать план и выйти, ничего не меняя |
| `-y` | `apply` | Не спрашивать подтверждения. **Обязателен** в неинтерактивном запуске |

`routineops help` печатает то же самое короче.

## Формат файла

```yaml
apiVersion: routineops/v1

groups:
  - name: Бухгалтерия
    color: "#3b82f6"
    software:                        # софт-правила именно этой группы
      - name: uTorrent
        rule: forbidden
      - name: 1C
        rule: allowed
    scriptPolicies:                  # какие скрипт-политики назначены группе
      - Еженедельная проверка дисков

scripts:
  - name: Проверка свободного места
    platform: linux                  # macOS | Windows | linux — ровно так, регистр важен
    file: scripts/proverka-svobodnogo-mesta.sh   # путь относительно YAML
  - name: Однострочник
    platform: linux
    content: |                       # альтернатива file для мелочи
      df -h /

scriptPolicies:
  - name: Еженедельная проверка дисков
    script: Проверка свободного места   # ссылка по ИМЕНИ, не по id
    trigger: schedule                    # schedule | event_trigger | on_connect
    schedule:
      cron: "0 9 * * 1"                  # обязателен при trigger: schedule
    active: true
```

Значения `platform` и `trigger` уезжают на сервер как есть и проверяются им: словарь
платформ — `macOS`, `Windows`, `linux` (историческая разнобойность регистра, `Linux`
сервер не примет), триггеров — `schedule`, `event_trigger`, `on_connect`. Скрипт для
`linux` доступен и на macOS-устройствах: shell-семейство считается общим, отдельно
стоит только Windows с PowerShell.

Тела скриптов лежат отдельными файлами, а не инлайном: так diff в git читается
построчно, работает подсветка синтаксиса, и правка скрипта не переписывает YAML
целиком. `file` и `content` взаимоисключающи — заполнено ровно одно.

Ключи `schedule` и `event` уезжают на сервер как есть: формат триггера задаёт сервер,
и CLI не должен разъезжаться с ним на каждое новое поле.

### Идентичность — имя

Ресурс опознаётся по имени, а не по id: id в YAML не попадают вовсе. Поэтому файл
переносим между инсталляциями и читается человеком в git-диффе.

Обратная сторона: имена уникальны (без учёта регистра и краевых пробелов) на уровне
базы — миграция 026 для групп, 033 для скриптов и политик. Попытка создать второй
объект с занятым именем вернёт `409`. Переименование в файле = создание нового
объекта; старый останется на сервере нетронутым (см. правило скоупа ниже).

## Что apply делает с сервером

**Файл управляет только тем, что в нём названо.** Группа, скрипт или политика, не
упомянутые в YAML, не трогаются — `apply -f accounting.yaml` не должен снести
настройки соседнего отдела. Никаких удалений «всего, чего нет в файле», CLI не
делает.

**Внутри названной группы перечисленные коллекции авторитетны.** Софт-правило, которого
нет в файле, будет удалено, а лишнее назначение политики — снято. Иначе YAML перестал
бы описывать реальность, и убрать правило через него стало бы невозможно.

**Отсутствующий ключ ≠ пустой список.** Нет ключа `software` — аспект не управляется,
правила на сервере остаются как есть. `software: []` — правил быть не должно, все
существующие снимаются. То же для `scriptPolicies`.

**Порядок шагов:** скрипты → скрипт-политики → группы. Политика ссылается на скрипт, а
группа — на политику; обратный порядок означал бы «политика на несуществующий скрипт».

**План печатается до применения**, и каждый выполненный шаг отмечается на экране. На
первой же ошибке apply останавливается и сообщает, сколько шагов успело примениться:
наполовину применённый план — состояние, в котором оператор обязан разобраться, а не
узнать о нём из общего «не получилось».

Изменение расписания политики выполняется правкой на месте (`PUT`), а не
пересозданием: у пересозданной политики был бы новый id, и все назначения на группы
отвалились бы.

## Рецепты

**Конфигурация в git.** Заведите отдельный репозиторий (или каталог) с
`routineops.yaml` и `scripts/`. Настроили руками в интерфейсе — выгрузили, посмотрели
diff, закоммитили. Выгрузка стабильна: всё сортируется по имени, поэтому повторный
`export` без изменений на сервере даёт пустой diff.

**Проверка в CI.** `apply` в неинтерактивном запуске требует явного `-y`, иначе
завершится ошибкой: автоматика не должна раскатывать правку конфигурации парка молча.
Полезная пара шагов — `-dry-run` на pull request и `-y` на merge.

**Перенос на другую инсталляцию.** Выгрузите с одного сервера, примените на другом:
id нигде не участвуют, ссылки идут по именам. Устройства при этом не переезжают —
их подключают энроллментом.

## Ограничения

- Нет удаления объектов: убрать группу, скрипт или политику можно только в интерфейсе.
  Это следствие правила скоупа — файл не знает, что на сервере лишнее, он знает лишь,
  что должно быть.
- Нет управления устройствами, пользователями и лицензией.
- Софт-правила без группы (глобальные, на конкретное устройство) невидимы для CLI:
  `export` их пропускает, `apply` не трогает.
- **Фильтр платформ у софт-правил в YAML не выражается.** Правило, ограниченное в
  интерфейсе, скажем, только Windows, выгрузится как обычное `name`/`rule`, а
  созданное по файлу правило действует на все платформы. Существующие правила при
  этом не переписываются: apply сверяет имя и тип, поэтому уже настроенный фильтр
  переживает применение — теряется он только при переносе на другой сервер или при
  смене типа правила (`allowed` ⇄ `forbidden` выполняется пересозданием).
- `apply` не транзакционен: план применяется шагами, откат не делается.
